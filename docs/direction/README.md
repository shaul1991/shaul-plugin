# 플러그인 개발 방향성 기록 (Direction Log)

이 디렉토리는 `project-lifecycle` 플러그인의 **개발 방향성·요구사항·아키텍처 결정**을 시간순으로 보존하는 곳이다. 단발성 변경 노트(CHANGELOG)나 출시 준비 계획(`docs/plans/`)과 달리, 여기 쌓이는 문서는 **"왜 플러그인이 이런 모습으로 만들어졌는가"**라는 질문에 답하는 영구 레퍼런스다.

## 무엇을 기록하는가

- 사용자(저장소 오너)가 제시한 요구사항·원칙(원문 그대로 보존)
- 그 요구를 충족시키기 위한 설계 결정과 그 근거
- 미래 변경 시 지켜야 할 불변 원칙
- 결정의 영향을 받는 출시 버전(혹은 출시 예정 버전)

## 무엇은 기록하지 않는가

- 일반적인 변경 이력 — `CHANGELOG.md`로 간다
- 일회성 작업 계획·리허설 — `docs/plans/`로 간다
- 코드의 *현재 상태* 설명 — README와 SKILL.md 본문으로 충분하다

## 파일 컨벤션

| 항목 | 규칙 |
|------|------|
| 파일명 | `YYYY-MM-DD-<주제-슬러그>.md` (`kebab-case`) |
| 위치 | `docs/direction/` 직속 (하위 폴더 없음) |
| 언어 | 한국어 (CHANGELOG와 동일) |
| 길이 | 1–3 페이지 권장. 길어지면 별도 파일로 분리 |

## 권장 문서 구조

```markdown
# <주제>

- **작성일**: YYYY-MM-DD
- **반영 출시**: vX.Y.Z (또는 "예정")
- **관련 PR/커밋**: <링크 또는 커밋 해시>
- **상태**: Active / Superseded by <링크> / Deprecated

## 사용자 원문 요구사항

> (사용자가 보낸 메시지를 가능한 한 원문 그대로 인용. 시간이 지나도
> "당시 요구가 정확히 무엇이었는가"를 추적할 수 있도록 보존한다.)

## 도출한 원칙

1. ...
2. ...

## 설계 결정

| ID | 결정 | 근거 |
|----|------|------|
| D1 | ... | ... |

## 미래 변경 시 지킬 것

- ...

## 관련 문서

- CHANGELOG `[vX.Y.Z]` 항목
- `<관련 SKILL.md / agent.md 경로>`
```

## 새 항목 추가 절차

1. 위 컨벤션에 맞춰 새 파일을 생성한다.
2. 사용자 원문 요구사항을 가능한 한 원본 그대로 보존한다(요약·의역 최소화).
3. 도출한 원칙·설계 결정·미래 변경 시 지킬 것을 명시한다.
4. 본 README의 인덱스 표 맨 위에 한 줄을 추가한다(시간 역순 유지).
5. 결정이 후속 문서로 대체될 경우, 기존 문서의 "상태"를 `Superseded by ...`로 갱신한다 — 삭제하지 않는다(이력은 보존).
