# 아키텍처 패턴 가이드

## 애플리케이션 아키텍처 패턴

### 모놀리식 (Monolithic)
- **적합**: 소규모 팀, MVP, 단순한 도메인
- **장점**: 배포 단순, 디버깅 용이, 트랜잭션 관리 쉬움
- **단점**: 스케일링 한계, 배포 결합도 높음
- **권장 시점**: 팀 5명 이하, 사용자 10만 이하

### 마이크로서비스 (Microservices)
- **적합**: 대규모 팀, 독립 배포 필요, 다양한 확장 요구
- **장점**: 독립 배포, 기술 다양성, 개별 스케일링
- **단점**: 운영 복잡도, 분산 트랜잭션, 네트워크 오버헤드
- **권장 시점**: 팀 10명 이상, 명확한 도메인 경계 존재

### 모듈러 모놀리스 (Modular Monolith)
- **적합**: 중간 규모, 향후 마이크로서비스 전환 고려
- **장점**: 모놀리식의 단순함 + 모듈 간 경계 명확
- **단점**: 모듈 경계 유지에 규율 필요
- **권장 시점**: 대부분의 MVP — 가장 실용적 선택

## API 스타일

| 스타일 | 적합한 상황 | 주의점 |
|--------|-----------|--------|
| REST | 범용, CRUD 중심, 공개 API | 오버/언더 페칭 가능 |
| GraphQL | 다양한 클라이언트, 유연한 쿼리 | 학습 곡선, N+1 문제 |
| gRPC | 서비스 간 통신, 고성능 | 브라우저 직접 호출 제한 |
| WebSocket | 실시간 양방향 통신 | 상태 관리 복잡 |

## 데이터베이스 선택

| 유형 | 제품 예시 | 적합한 상황 |
|------|----------|-----------|
| RDBMS | PostgreSQL, MySQL | 구조화된 데이터, 복잡한 관계, 트랜잭션 |
| Document DB | MongoDB, DynamoDB | 유연한 스키마, 읽기 중심, 빠른 반복 |
| Key-Value | Redis, Memcached | 캐싱, 세션, 실시간 카운터 |
| Search | Elasticsearch | 전문 검색, 로그 분석 |
| Time-series | InfluxDB, TimescaleDB | 시계열 데이터, 모니터링 |

## 프론트엔드 아키텍처

| 패턴 | 적합한 상황 |
|------|-----------|
| SPA (Single Page App) | 복잡한 인터랙션, 앱 같은 경험 |
| MPA (Multi Page App) | SEO 중요, 콘텐츠 중심 |
| SSR + Hydration | SPA + SEO 모두 필요 (Next.js, Nuxt) |
| Static Site + API | 블로그, 문서 사이트 |

## 인프라 패턴

| 패턴 | 적합한 상황 | 비용 특성 |
|------|-----------|----------|
| Serverless (Lambda 등) | 이벤트 기반, 가변 트래픽 | 사용량 비례 |
| Container (ECS/K8s) | 안정적 트래픽, 제어 필요 | 기본 비용 있음 |
| PaaS (Railway, Fly 등) | MVP, 인프라 관리 최소화 | 중간 |
| VM (EC2, GCE) | 레거시, 특수 요구 | 고정 비용 |
