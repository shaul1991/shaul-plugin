# ALM (Application Lifecycle Management) 가이드

## ALM이란?

Application Lifecycle Management는 소프트웨어 애플리케이션의 전체 수명 동안 요구사항, 개발, 테스트, 배포, 유지보수를 통합적으로 관리하는 체계.

핵심 목표: **추적성(Traceability)**, **투명성(Transparency)**, **반복성(Repeatability)**

## ALM의 3대 영역

### 1. Governance (거버넌스)
의사결정 체계와 품질 게이트:

| 게이트 | 위치 | 확인 사항 |
|--------|------|----------|
| Idea Gate | Phase 1 → 2 | 아이디어가 투자할 가치가 있는가? |
| Scope Gate | Phase 2 → 3 | MVP 스코프가 현실적인가? |
| Design Gate | Phase 3 → 4 | 기술 결정이 합리적인가? |
| Build Gate | Phase 5 → 6 | 코드 품질이 충분한가? |
| Release Gate | Phase 7 → 배포 | 릴리즈 기준을 충족하는가? |

각 게이트에서 "Go / No-Go / Iterate" 판단.

### 2. Development (개발)
실제 산출물을 만드는 활동:

- 요구사항 관리 (Phase 2)
- 설계 (Phase 3-4)
- 구현 (Phase 5)
- 빌드 & 배포 (Phase 6)

### 3. Operations (운영)
배포 후 지속적 활동:

- 모니터링 & 알림
- 인시던트 관리
- 성능 최적화
- 사용자 피드백 수집 → 다음 이터레이션

## 추적성 매트릭스 (Traceability Matrix)

요구사항에서 테스트까지의 추적:

```
요구사항 ID → 유저 스토리 ID → 설계 문서 → 코드 모듈 → 테스트 케이스
F-001      → US-001, US-002 → api-spec   → auth/     → auth.test.ts
```

이를 통해:
- 요구사항이 빠짐없이 구현되었는지 확인
- 변경 시 영향받는 코드/테스트 식별
- 테스트 커버리지의 요구사항 기반 평가

## 변경 관리 프로세스

```
변경 요청 → 영향 분석 → 승인/거부 → 구현 → 검증 → 기록
```

### 변경 요청서 템플릿
```markdown
## 변경 요청
- **요청자**: 
- **날짜**: 
- **유형**: 기능 변경 / 버그 수정 / 개선 / 기술 부채
- **설명**: 
- **사유**: 
- **영향 범위**: 
  - 산출물: 
  - 코드: 
  - 테스트: 
- **우선순위**: Critical / High / Medium / Low
- **예상 공수**: 
```

## 반복(Iteration) 관리

ALM은 Waterfall이 아닌 반복적 접근을 권장:

```
Iteration 1: Phase 1-7 (MVP)
  └─ 피드백 수집
Iteration 2: 개선 사항 반영
  └─ Phase 2부터 재진행 (스코프 조정)
Iteration 3: 확장 기능
  └─ 필요한 단계만 선택적으로
```

각 이터레이션마다:
1. 이전 이터레이션의 회고(Retrospective) 수행
2. 학습 사항을 다음 이터레이션에 반영
3. lifecycle.md에 이터레이션 이력 기록

## 프로젝트 건강도 지표

### 정량적 지표
| 지표 | 건강 | 주의 | 위험 |
|------|------|------|------|
| 테스트 커버리지 | >80% | 60-80% | <60% |
| 빌드 성공률 | >95% | 80-95% | <80% |
| 평균 PR 리뷰 시간 | <24h | 24-48h | >48h |
| 오픈 버그 수 | <10 | 10-30 | >30 |
| 기술 부채 비율 | <15% | 15-30% | >30% |

### 정성적 지표
- 팀의 자신감 수준 (릴리즈에 대한 확신)
- 문서의 최신성 (마지막 업데이트가 1주 이내)
- 의존성 최신성 (major 버전 뒤쳐진 패키지 수)
