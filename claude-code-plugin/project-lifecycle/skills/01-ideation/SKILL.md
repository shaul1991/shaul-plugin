# Phase 1: 아이디어 수집 (Ideation)

프로젝트의 출발점. 문제를 정의하고, 아이디어를 수집·정제하여 프로젝트로 발전시킬 가치가 있는지 검증한다.

## 필수: Plan → Review → Execute → Re-verify

**이 Phase를 시작하기 전에 반드시 거버넌스 프로세스를 따른다.**

1. **PLAN** — 실행계획서를 작성한다 (`governance` 스킬의 `references/execution-plan-template.md` 참조)
   - 해당 Phase의 `docs/` 디렉토리에 `execution-plan.md`로 저장
   - 목표, 범위, 실행 단계, 성공 기준을 구체적으로 기술
2. **REVIEW** — 실행계획서를 사용자에게 제시하고 명시적 수락을 받는다
   - 승인(Approved) → 실행 절차로 진행
   - 수정 요청(Revise) → 계획 수정 후 재검증
   - 거부(Rejected) → 근본적 재설계
3. **EXECUTE** — 수락된 계획에 따라 아래 실행 절차를 수행한다
4. **RE-VERIFY** — 실행 완료 후 산출물과 결과를 재검증한다
   - 성공 기준 대비 달성 여부 확인
   - 산출물 완결성 및 이전 Phase와의 정합성 검증
   - 교훈(Lessons Learned) 기록
   - 통과(Pass) → 다음 Phase 진행 / 미달(Fail) → 보완 실행 또는 계획 재수립

> ⚠️ 실행계획 수립과 수락 없이 실행에 들어가지 않는다. 실행 후 재검증 없이 다음 Phase로 넘어가지 않는다.

## 실행 절차

### Step 1: 문제 정의
사용자에게 다음을 확인한다:

1. **해결하려는 문제** — 누구의, 어떤 고통(pain point)인가?
2. **대상 사용자** — 누가 이 문제를 겪고 있는가?
3. **현재 대안** — 지금은 어떻게 해결하고 있는가?
4. **왜 지금?** — 이 시점에 이 문제를 풀어야 하는 이유

### Step 2: 아이디어 브레인스토밍
사용자의 초기 아이디어를 받아 아래 프레임워크로 확장한다:

- **How Might We (HMW)**: "어떻게 하면 ~할 수 있을까?" 형태로 문제를 재정의
- **Crazy 8s**: 8개의 빠른 솔루션 스케치 (텍스트 기반)
- **역발상**: 가장 나쁜 해결책을 먼저 생각하고 뒤집기

### Step 3: 아이디어 필터링
수집된 아이디어를 평가 매트릭스로 필터링한다:

| 평가 기준 | 가중치 | 설명 |
|-----------|--------|------|
| 사용자 임팩트 | 30% | 문제 해결 시 사용자에게 미치는 영향 |
| 실현 가능성 | 25% | 기술적, 리소스적 구현 가능 여부 |
| 시장 규모 | 20% | 타겟 시장/사용자 규모 |
| 차별화 | 15% | 기존 대안 대비 차별점 |
| 시급성 | 10% | 지금 해야 하는 이유의 강도 |

### Step 4: 산출물 생성
평가가 끝나면 프로젝트 루트에 아래 파일을 생성한다:

**`docs/01-ideation/idea-brief.md`** 파일을 `references/idea-brief-template.md` 템플릿 기반으로 작성.

## 가이드라인

- 아이디어 단계에서 기술 스택을 확정하지 않는다 — 문제와 사용자에 집중
- 최소 3개 이상의 대안 아이디어를 비교한 뒤 선택한다
- "왜?"를 5번 반복하여 근본 원인에 도달한다 (5 Whys)
- 경쟁사/대안 분석은 간략하게 — 심화 분석은 기획 단계에서
